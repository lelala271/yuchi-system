# 安全设计

本文档解决的问题是：这个项目为什麽不能只停留在“能跑”，而必须把鉴权、默认密码、实时链路和主题权限一起收紧。

一句话结论：智能车平台最危险的部分往往不是页面，而是实时链路；如果 WebSocket、MQTT 和默认账号不收口，系统越方便越容易失控。

## 1. 风险面总览

| 风险点 | 如果不处理会怎样 |
| --- | --- |
| 硬编码 JWT 密钥 | 任何拿到代码的人都能伪造 token |
| 默认弱口令管理员 | 演示环境很容易被直接接管 |
| 开放注册 | 任意人都可能创建新用户 |
| WebSocket URL 带 token | token 更容易进入日志或中间层记录 |
| MQTT 匿名连接 | 未授权设备可接入消息总线 |
| MQTT 主题无限制 | 设备或页面可能越权发消息和订阅 |
| 无接口限流 | 登录接口容易被暴力尝试 |
| 无负载限制 | 超大消息可能压垮单机服务 |

## 2. 当前已经收紧的设计

### 2.1 运行时密钥

`backend/config.js` 会在首次启动时生成运行时密钥文件，而不是把关键秘密写死在源码中。

自动生成内容包括：

- `AUTH_TOKEN_SECRET`
- 初始管理员密码
- MQTT 设备密码

这意味着源码仓库本身不需要携带固定秘密。

### 2.2 注册默认关闭

`ALLOW_REGISTRATION` 默认是 `false`，而不是对外开放注册。

这条配置的意义是：

- 演示环境默认更安全
- 生产环境默认更符合内部平台管理习惯
- 不会因为前端留了注册入口就自动放开用户创建

### 2.3 强密码规则

后端密码校验要求：

- 长度 `12-128`
- 同时包含大写字母
- 同时包含小写字母
- 同时包含数字
- 同时包含特殊字符

密码强度是在后端执行的，不依赖前端提示。

### 2.4 历史弱口令自动轮换

`backend/user-store.js` 会检测历史弱口令管理员账号，并自动轮换成运行时密钥文件里的新密码。

这解决的是“老数据库被沿用，但旧管理员密码过弱”的现实问题。

### 2.5 WebSocket 鉴权收紧

WebSocket 首选从 `Sec-WebSocket-Protocol` 中读取 `auth.<token>`，而不是要求用户把 token 放在 URL 查询参数里。

这样做的原因：

- 减少 token 落到 URL 日志里的概率
- 明确把鉴权和协议协商绑定起来
- 便于前端统一管理

### 2.6 MQTT 双身份模型

MQTT 客户端分两类：

| 客户端类型 | 认证方式 | 权限定位 |
| --- | --- | --- |
| 页面应用客户端 | `username + JWT` | 只能订阅指定主题，不能发布设备消息 |
| 设备客户端 | `MQTT_DEVICE_USERNAME + MQTT_DEVICE_PASSWORD` | 可以按白名单发布状态并订阅控制主题 |

这条分层很关键，因为页面不是设备，设备也不是页面。

### 2.7 主题白名单

设备发布主题必须匹配：

- `yuchi/vehicle/{vehicleId}/status`
- `yuchi/network/metrics`
- `yuchi/perception/yolo`
- `yuchi/perception/lane`
- `yuchi/v2x/bsm`
- `yuchi/v2x/spat`
- `yuchi/twin/state`

设备订阅主题必须匹配：

- `yuchi/vehicle/+/control`
- `yuchi/vehicle/{vehicleId}/control`

应用客户端只能订阅平台允许的固定主题集合。

### 2.8 限流和负载上限

| 控制项 | 作用 |
| --- | --- |
| API 全局限流 | 防止普通接口被滥用 |
| 登录限流 | 防止暴力尝试账号密码 |
| HTTP Body 限制 | 防止超大请求体 |
| WebSocket Payload 限制 | 防止异常大消息 |
| MQTT Payload 限制 | 防止 Broker 被大包拖垮 |
| WebSocket 最大连接数 | 防止单机连接被撑爆 |

## 3. 安全边界该怎么理解

| 组件 | 当前边界 |
| --- | --- |
| HTTP API | 认证接口开放，业务接口需要 JWT |
| WebSocket | 需要合法 token，且要通过 Origin 检查 |
| MQTT 页面连接 | 必须带页面用户名和 JWT |
| MQTT 设备连接 | 必须带设备专用口令 |
| 用户注册 | 默认关闭 |
| 初始管理员 | 自动生成强密码，首次登录应立即修改 |

## 4. 上线前必须做的安全动作

1. 手工设置生产环境 `AUTH_TOKEN_SECRET`。
2. 检查 `ALLOW_REGISTRATION=false`。
3. 确保 `ENABLE_SIMULATOR=false`。
4. 更换初始管理员密码。
5. 限制 `runtime-secrets.json` 的文件访问权限。
6. 只给设备下发设备专用 MQTT 凭据。
7. 配置好 `CORS_ORIGINS`，不要放任任意来源。

## 5. 常见误区

| 误区 | 正确理解 |
| --- | --- |
| 只要前端没显示注册按钮就安全 | 错，后端必须默认禁用注册 |
| MQTT 在内网就不用认证 | 错，内网设备同样可能误接、串接或被滥用 |
| WebSocket 只是看数据，不需要严格鉴权 | 错，实时流里也包含敏感状态和控制反馈 |
| 初始管理员密码只在本机测试用 | 错，很多事故就是测试配置流入正式环境 |

## 6. 最终记忆表

| 维度 | 当前策略 |
| --- | --- |
| 账号安全 | 强密码、改密接口、弱口令轮换 |
| token 安全 | 运行时密钥、JWT 校验 |
| 页面实时安全 | WebSocket 鉴权 + Origin 校验 |
| 设备实时安全 | MQTT 强认证 + 主题白名单 |
| 服务可用性 | 限流、连接数限制、消息大小限制 |
| 默认策略 | 尽量收紧，而不是默认放开 |
